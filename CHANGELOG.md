# Changelog

All notable user-facing changes to Tardigrade are documented here.

## [0.4.0] - 2026-07-03

### Performance
- **Idle keepalive connections no longer occupy a worker thread (#138)** — Tardigrade serves requests on a bounded worker pool with blocking I/O. Previously a worker stayed attached to a keepalive connection for its entire lifetime (blocked in `read()` between requests), so once concurrent connections exceeded the worker count the surplus connections starved and tail latency exploded. After serving a request, a keepalive connection is now parked off the worker pool: its state (fd, pooled session, optional TLS session) moves into a registry, its fd is watched by the event loop, and the worker returns to the pool. When the next request arrives the event loop hands the connection back to a worker, which serves one request and re-parks or closes. Active request handling still uses a worker with blocking I/O, so only the idle gap is decoupled. Measured on the 4-core deployment hardware at the default `worker_threads` = CPU count: **p90 dropped from 26.5 ms to 1.33 ms at 10 connections (20×), and from 412 ms to 6.2 ms at 100 connections (66×)**, with throughput up 10–27%. TLS connections are parked too (the OpenSSL session moves with the connection and `SSL_pending` buffered/pipelined data is served immediately); HTTP/2 is intentionally not parked (it multiplexes internally). New Prometheus metrics: `tardigrade_keepalive_parked_connections`, `tardigrade_keepalive_resumes_total`, `tardigrade_keepalive_timeouts_total`, `tardigrade_keepalive_closed_total`. With parking, sizing `worker_threads` to CPU count is correct again; the prior interim guidance to raise it to ~peak concurrent connections is only needed until this is deployed.
- **Interim worker-thread sizing for the homelab edge (stopgap for #138)** — the blocking thread-per-connection model needs `worker_threads` ≥ peak concurrent connections to keep tail latency low, but the default sized it to CPU count (4 on the deploy box). The homelab deploy now pins `TARDIGRADE_WORKER_THREADS=64`, cutting the under-load p90 from ~26 ms to sub-millisecond before the durable keepalive-parking fix lands. `worker_threads` defaulting to CPU count is documented as incorrect for the current blocking model in `edge_gateway.zig`.

### Fixed
- **HTTP/2 requests no longer fail with `Http2CompressionError`, and h2 responses are now protocol-valid (#138)** — the HPACK decoder rejected every Huffman-encoded header string (`error.HuffmanUnsupported`), so any real HTTP/2 client (curl/nghttp2, browsers, h2load) — which Huffman-encode header fields by default — got a `GOAWAY(COMPRESSION_ERROR)` and an empty body before the request was ever served. Added a full RFC 7541 §5.2 / Appendix B Huffman decoder (`src/http/hpack_huffman.zig`, validated against the RFC C.4/C.6 test vectors) wired into `decodeStringAlloc`. Fixing decode then surfaced a second, previously-masked bug: the h2 response path emitted mixed-case `X-Request-ID` / `X-Correlation-ID` field names, which HTTP/2 forbids (RFC 7540 §8.1.2 requires lowercase), causing compliant clients to reject the response — those names are now lowercased before HPACK encoding. Verified end-to-end with curl `--http2`, `nghttp`, and `h2load` (200/200 requests, 0 protocol errors).
- **Proxied requests no longer return spurious `502`s when the upstream closes an idle keep-alive connection** — under sustained load, an upstream can close a pooled keep-alive connection between the moment Tardigrade selects it and the moment it would serve the request, which surfaces as `error.HttpConnectionClosing` (zero response bytes received). Because the request was never delivered, Tardigrade now transparently retries it on a fresh connection (up to 2 extra attempts), independent of the configured `TARDIGRADE_UPSTREAM_RETRY_ATTEMPTS` budget and without counting the close against upstream health / circuit-breaker state. The retry is gated by the existing idempotent-only guard, so non-idempotent methods (`POST`/`PATCH`) are only retried this way when `TARDIGRADE_UPSTREAM_RETRY_IDEMPOTENT_ONLY=false`. In a loopback proxy benchmark this cut the under-load `502` rate from ~1% (4,342 of ~438k requests) to 0.006% (28, all unrelated mid-stream `ReadFailed`) with no change to throughput or tail latency. Added 4 unit tests.
- **Homelab BearClaw edit pages no longer trip the generic buffered-upstream cap** — the Blink-managed homelab start and rollback paths now pin `TARDIGRADE_MAX_BUFFERED_UPSTREAM_RESPONSE_BYTES=4194304` in `tardigrade.env`, and verify that the override is present after deploy. This keeps large BearClaw dashboard editor responses from failing at the edge with `{"code":"upstream_error","message":"Upstream connection failed"}` even though the upstream Rails app is healthy.
- **Stop command now kills all tardigrade processes, not just the systemd service unit** — `pkill -x tardigrade` is appended to the blink.toml stop command so orphaned processes started outside systemd (e.g. from a previous manual deploy) are always cleaned up before a new instance starts. Previously two instances could share port 8443 via SO_REUSEPORT, causing intermittent upstream errors depending on which instance handled each request.

- **Proxied responses from gzip-capable upstreams no longer arrive garbled in the browser** — `Accept-Encoding` is now stripped from upstream requests so upstreams always send uncompressed bodies to Tardigrade. Previously the client's `Accept-Encoding: gzip` was forwarded to the upstream, which sent a gzip body, but Tardigrade unconditionally stripped the `Content-Encoding: gzip` response header before forwarding — leaving the browser with binary data it had no idea was compressed. Tardigrade is responsible for its own encoding negotiation with upstreams.
- **Buffered proxy responses no longer corrupt large upstream bodies at the 8 KiB scratch-buffer boundary (#135)** — when a proxied response head plus the first body chunk overflowed Tardigrade's internal response scratch buffer, the buffered write path replayed the entire upstream body instead of only the unwritten suffix. Large CSS, image, font, and other buffered proxy responses now preserve the upstream bytes exactly. Closes #135.
- **Proxy keep-alive recovery now drops dead pooled upstream sockets after backend replacement, and zero-length POST proxying no longer panics (#134)** — when a reused upstream keep-alive connection dies during a backend restart, Tardigrade now marks that socket non-reusable so later requests reconnect without requiring a gateway restart. Proxy requests that send an explicit empty POST body now emit a correct `Content-Length: 0` upstream request instead of tripping the raw proxy path assertion. Closes #134.
- **Workflow shell steps no longer inline GitHub expressions, and the remaining superfluous-action alert is gone (#117)** — the last direct `${{ ... }}` expansions inside `run:` commands now flow through `env:` first, including the release checksum verification path and container digest export path. The workflow set no longer relies on the previously flagged redundant action pattern, so the outstanding `zizmor` template-injection and superfluous-action drift is resolved. Closes #117.
- **Workflow artifact uploads now stage explicit temp-path outputs before `upload-artifact` (#116)** — CI, release, container, and scorecard workflows now copy intermediate files into dedicated temp artifact paths, upload exact files instead of workspace-relative globs, and set explicit retention windows for transient artifacts. This removes the remaining `artipacked`-style credential-persistence risk from GitHub Actions artifact handling while keeping release publishing and multi-job handoff behavior intact. Closes #116.
- **GitHub Actions token defaults now stay least-privilege across release, container, and scorecard workflows (#115)** — the remaining workflow-scope broad read default in `.github/workflows/scorecard.yml` now matches the repo policy of `permissions: contents: read`, while jobs that publish SARIF, tags, releases, or container images keep only their explicit job-level elevation. This aligns the workflow set with the intended least-privilege `GITHUB_TOKEN` model and clears the remaining permissions-policy drift. Closes #115.
- **Operator CLI now exposes `status` and `print-config`, and validation output is more actionable (#112)** — the CLI help text now documents diagnostic commands explicitly, `tardigrade validate` prints the resolved config path plus an effective-config summary on success, `tardigrade status` reports process state when a pid target is available, and `tardigrade print-config` / `tardigrade config print` expose a compact effective runtime summary without starting the gateway. The parser now prints clearer command-line errors before usage text. Closes #112.
- **Benchmark baselines now capture latency percentiles plus target CPU/RSS and regress on more than req/s (#110)** — `benchmarks/run.sh` now records p50/p95/p99/p999 per scenario and, when pointed at the live Tardigrade PID, samples average CPU and peak RSS into the saved JSON. Markdown benchmark reports now surface those fields, the CI smoke harness validates the richer artifact contract, and release/perf docs now describe how to capture reproducible process metrics alongside throughput. Closes #110.
- **Observability model now includes latency histograms and consistent request IDs in structured logs (#109)** — `/status/metrics` now exports a low-overhead global `tardigrade_request_latency_ms` histogram alongside the existing connection, worker, and event-loop gauges, and runtime JSON logs now emit `request_id` wherever `correlation_id` is present so component logs align with access-log fields. Added `docs/OBSERVABILITY.md` to describe the combined log/metric/trace contract, including `traceparent` propagation on proxied requests. Closes #109.
- **Connection model and worker-pool observability are now explicit (#108)** — documented the current runtime architecture as a non-blocking main-thread accept loop plus a bounded blocking worker pool rather than an async/hybrid request engine, and exported worker-pool gauges for active jobs, queued jobs, configured worker threads, and queue capacity through `/status/metrics`. This makes the chosen event-loop strategy visible to operators and benchmark runs without a broad concurrency rewrite. Closes #108.
- **Runtime allocator policy no longer uses `DebugAllocator` on the gateway hot path (#106)** — the `run` path now uses Zig 0.16's shared `std.heap.smp_allocator` for long-lived runtime state and HTTP/2 connection bookkeeping, while one-shot CLI/control-plane commands keep `DebugAllocator` leak diagnostics. Request scratch still flows through arenas and pooled buffers, and the engineering guide/agent docs now spell out the allocator ownership model explicitly. Closes #106.
- **Core v1 support contract is now explicit (#105)** — added `docs/SUPPORT_MATRIX.md` to define the stable Core v1 scope and classify the visible feature surface as `stable`, `experimental`, `adapter`, or `internal`. Updated `README.md` and `AGENTS.md` to point at the matrix, marked the BearClaw example/docs as example-scoped rather than a blanket production promise, and added a release-checklist reminder to update the support matrix when public behavior or maturity claims change. Closes #105.
- **Host-native release/install hardening (#86)** — Release archives now ship the operator-facing docs alongside the binary, alias, and license; the release workflow publishes `install.sh`, generates checksums for the install/bootstrap assets, and verifies the checksum manifest before publishing. `scripts/install.sh` now fails closed on missing or mismatched checksums, supports smoke-testable release mirrors, and stages installs via temp paths before moving them into place. Added `scripts/test-install.sh` and `scripts/test-deb-package.sh`, a CI packaging-smoke job, a packaged starter config, and DEB smoke coverage that validates the binary path, systemd unit, config/env files, ownership, and uninstall behavior. Closes #86.
- **Canonical perf target route set drifted from the benchmark harness (#100)** — the benchmark target only exposed `/proxy/payload-64k.bin` by default even though the harness and upstream fixture had already grown a `256 KiB` scenario. The canonical release-baseline metadata and docs now match the benchmark target route set, `release-baseline.sh` includes both payload-size scenarios by default, and the upstream fixture remains the required `tardigrade-upstream.service` backend for canonical runs. Closes #100.
- **Buffered proxy response size was coupled to inbound request parsing (#99)** — the plain HTTP/1 buffered proxy readers shared `MAX_REQUEST_SIZE` (`256 KiB`) with request parsing, so operators could not raise proxied response payload ceilings without also changing the parser constant in code. Added `TARDIGRADE_MAX_BUFFERED_UPSTREAM_RESPONSE_BYTES` (default `262144`) as a dedicated buffered-upstream limit, switched the HTTP/1 proxy readers to use it, and made oversized upstream bodies fail deterministically with a stable `502` path instead of relying on truncated partial reads. Added config validation, 1 unit test, and 3 integration tests. Closes #99.
- **CSP and X-Frame-Options missing from API-preset responses (#96)** — `SecurityHeaders.api` explicitly zeroed `content_security_policy` and `x_frame_options`, so any response rendered through the API preset (proxy pass, versioned API routes, error replies) silently omitted `Content-Security-Policy` and `X-Frame-Options`. Removed the overrides so `api` is now identical to `default`: every response carries `Content-Security-Policy: default-src 'self'` and `X-Frame-Options: DENY`. Operators who genuinely need a looser policy can override individual fields in config. Updated 1 unit test. Closes #96.
- **Server header disclosed version string (#97)** — All 5 `Server` header emission sites in `edge_gateway.zig` and both sites in `response.zig` emitted `Server: tardigrade/<version>`, leaking the precise release tag to unauthenticated clients. Removed the version suffix; header now reads `Server: tardigrade`. Updated 2 unit tests. Closes #97.

### Refactored
- **Split response writing and target resolution out of `gateway_proxy.zig` (#182)** — response serialization helpers now live in `src/gateway_proxy_response.zig`, and proxy target/query/redirect resolution helpers live in `src/gateway_proxy_target.zig`. `gateway_proxy.zig` keeps compatibility re-exports for existing callers while focusing on bounded-buffered and streaming proxy transports.
- **Reduced gateway compatibility aliasing (#156)** — `src/edge_gateway.zig` now calls gateway subsystems through their owning modules instead of broad compatibility aliases, keeps only documented local runtime state shorthands, and moves remaining gateway helper tests into the modules that own the tested behavior. Closes #156.
- **Separated control-plane proxy helpers from the data-plane reverse-proxy path (#154)** — bounded JSON/API helper proxying now lives in `src/gateway_control_plane_proxy.zig`, while general `proxy_pass` traffic enters through `src/gateway_proxy_runtime.zig`. Fully materialized upstream-response helpers are explicitly named as bounded-buffer paths, and the data-plane runtime has a single compatibility executor ready for future streaming/backpressure work without routing through control-plane helpers.
- **Split focused gateway runtime modules out of `edge_gateway.zig` (#152)** — listener accept/overload rejection now lives in `src/gateway_accept.zig`, socket/proxy-protocol/request-read helpers in `src/gateway_connection.zig`, static file/error-page handling in `src/gateway_static_runtime.zig`, reload/maintenance/probe work in `src/gateway_shutdown.zig`, proxy runtime glue in `src/gateway_proxy_runtime.zig`, and route/handler/HTTP/3 dispatch glue in `src/gateway_handlers.zig`. `edge_gateway.zig` is reduced to runtime setup, connection lifecycle, and HTTP/2 framing while preserving existing behavior.
- **Internal runtime queues and Zig 0.16 list usage are now more consistent and cheaper to drain** — worker-pool dispatch queues, SSE event retention, HPACK dynamic-table eviction, and transcript recent-window retention now use O(1) front-pop structures instead of `orderedRemove(0)` shifting. The remaining source `std.array_list.Managed` call sites were converted to the repo’s unmanaged `.empty` style, and `zig_compat.zig` now initializes the global threaded `std.Io` runtime with synchronization plus a larger reader scratch buffer for the compat stream adapter. No public behavior or support-matrix contract changed.
- **Extracted `gateway_auth.zig` and `gateway_json_proxy.zig` from `edge_gateway.zig` (#103)** — continued the refactor: auth logic (`AuthResult`, `AuthFailureReason`, `authorizeRequest`, `resolveRequestConfig`, `evaluatePolicy`, `approvalPolicyError`, and all policy/device/token helpers) moved to `src/gateway_auth.zig`; JSON proxy execution (`ProxyResult`, `ProxyExecution`, `proxyJsonExecute`, `proxyJsonExecuteSingleAttempt`, `buildProxyCacheKey`, and cache-refresh helpers) moved to `src/gateway_json_proxy.zig`. `edge_gateway.zig` shrank from 5,925 to 4,653 lines. All 476 tests pass.
- **Extracted `gateway_proxy.zig` and `gateway_protocols.zig` from `edge_gateway.zig` (#103)** — continued the refactor: proxy transport types/execution (`UpstreamHeader`, `RawUpstreamResponse`, `executeRawHttpProxyRequest`, `executeUnixSocketHttpRequest`, `executeUpstreamHttpsWithMtls`, and ~30 response-writing/header utilities) moved to `src/gateway_proxy.zig`; FastCGI, SCGI, uWSGI, gRPC, mail, IMAP, and SMTP protocol handlers moved to `src/gateway_protocols.zig`. `edge_gateway.zig` shrank from 7,787 to 5,925 lines (24% further, 43% total reduction from 10,399). All 476 tests pass.
- **Extracted `gateway_state.zig` from `edge_gateway.zig` (#103)** — `src/edge_gateway.zig` shrank by 2,612 lines (25%, from 10,399 to 7,787) by extracting all runtime-state types (`GatewayState`, `WorkerContext`, `ReloadableConfigStore`, `ConnectionSessionPool`, and ~25 supporting types), the upstream selection and sticky-affinity helpers, the active-health config helper, and the proxy-cache disk helpers into a new focused module `src/gateway_state.zig`. `edge_gateway.zig` imports via const aliases so all existing call sites compile unchanged. Tests for `lcrngNext`, `selectUpstreamBaseUrl`, `selectUpstreamBaseUrlWeighted`, connection-session pool, and reloadable config store moved to the new module. No behavior changes; all 476 tests pass.

### Added
- **Bounded streaming reverse-proxy mode (#139)** — added `TARDIGRADE_PROXY_STREAMING_MODE=off|response|full` with a bounded `TARDIGRADE_PROXY_STREAM_BUFFER_SIZE` relay buffer so large upstream responses can stream without materializing the full body, and eligible fixed-length client uploads can relay without first fitting inside the request parser buffer. Streaming preserves filtered upstream headers, handles fixed-length/chunked/no-body/HEAD/204/304 responses, records proxy streaming/buffered byte and abort metrics plus upstream TTFB, and falls back to the bounded buffered compatibility path for retry-enabled routes, Unix-socket upstreams, custom upstream mTLS, rewrites, mirrors, and auth subrequests. Added live integration coverage for large downloads, chunked responses, no-body statuses, upstream truncation, and large uploads, plus benchmark scenarios for 1 MiB/16 MiB downloads and fixed-length upload testing. Closes #139.
- **Hot-path allocation regression budgets (#155)** — added `zig build bench-allocations`, an in-process benchmark that reports `allocations_per_request` and `bytes_allocated_per_request` for warm tiny static files, static 304s, warm proxy keep-alive helper work, and overload rejection. `zig build test` now enforces the same documented budgets so allocation-count and byte-count regressions fail with the scenario name and exceeded threshold. Closes #155.
- **Transfer throughput in benchmark JSON and reports (#102)** — `benchmarks/run.sh` now records `throughput_mbps` (MB/s) per scenario for wrk, h2load, and k6; fortio and scenarios where the tool does not report a transfer rate record `null`. `benchmarks/report.sh` adds a **MB/s** column to the markdown table, rendering `-` for older result files that predate the field. Backward-compatible: all existing saved JSON files continue to render correctly. Validation covered by the new `benchmarks/test-report.sh` script (8 tests).

- Verify tests TC-TARDIGRADE-005/006/007 — `security-header-csp`, `security-header-x-frame-options`, and `security-header-server-no-version` confirm that CSP, X-Frame-Options, and a version-free Server header are present on every deploy.
- Payload-size benchmark coverage for the canonical perf flow — `benchmarks/run.sh` now supports `proxy-payload-64k` and `proxy-payload-256k` scenarios plus path overrides, `benchmarks/release-baseline.sh` includes those scenarios by default unless overridden, and `benchmarks/fixtures/upstream_server.py` now serves both fixture payload sizes. The `256 KiB` case rides the existing exact-match guest route via `/proxy/payload-64k.bin?size=256k`, which keeps the canonical benchmark target compatible without depending on extra out-of-repo location blocks. ## [0.3.0] - 2026-05-11

### Fixed
- **Static file serving broken when `root` is set without `try_files` (top-level config)** — `serveTryFilesFallback` exited immediately when `try_files` was empty (`if (cfg.doc_root.len == 0 or cfg.try_files.len == 0) return error.NoTryFiles`), so operators who set only `root /path;` at the server level could not serve static files — all unmatched paths returned 404 even when the file existed in the webroot. Fixed by defaulting `effective_try_files` to `"$uri"` when `doc_root` is non-empty but `try_files` is empty, so the exact request path is checked in the webroot. Files that exist are served; files that do not exist still 404 correctly. The `location / { root ...; }` block form is unaffected. Added 1 unit test validating both the defaulting logic and the full file-serve path. 475 tests total. Closes #92.
- **TRACE method accepted on all routes (XST / ASVS-14.5.1)** — `handleConnection()` had no global TRACE guard; the TRACE method was forwarded to location blocks and proxy upstreams, enabling Cross-Site Tracing attacks. Added a pre-routing TRACE rejection that returns `405 Method Not Allowed` before any dispatch, mirroring the Host header check added in #90. Static-return location blocks (`return_response`) now also enforce GET/HEAD-only for non-redirect responses — `DELETE /health → 200` was silently succeeding and could mislead clients. Redirect responses (3xx) are method-agnostic and are unaffected. Added corpus case `tests/corpus/http/request/trace_method.http` and 2 unit tests. 473 tests total. Closes #88.
- **Upstream connection drop returned bare TCP close instead of 502 Bad Gateway** — when a proxied upstream closed the connection before sending a complete HTTP response, `parseRawUpstreamResponse` returned `error.UnsupportedHttpMethod` (misnamed) and `handleLocationProxyPass` propagated it unhandled, causing Tardigrade to close the TCP connection to the client with no response. All connection-error paths in the proxy retry loop now synthesise a proper `502 Bad Gateway` (or `504 Gateway Timeout` for timeout errors) using `sendApiError`, so clients always receive a complete HTTP message. Renamed the parse-error to `error.UpstreamProtocolError` for clarity. Added 3 unit tests for partial/empty upstream response detection. 471 tests total. Closes #94.
- **Correlation-ID spoofing and log poisoning via client-supplied X-Request-ID / X-Correlation-ID** — `isValid()` in `correlation_id.zig` accepted any alphanumeric string, allowing clients to inject arbitrary trace IDs into responses and access logs. Narrowed the validation to enforce the Tardigrade format `tg-<decimal>-<lowercase-hex>` exactly: IDs not matching this pattern are discarded and a fresh ID is generated. Prevents spoofing of trace IDs and poisoning of structured logs with operator-controlled values. 6 updated/new unit tests. 468 tests total. Closes #91.
- **HTTP/1.1 requests without Host header accepted (RFC 7230 §5.4 violation)** — `handleConnection()` routed requests before checking for the Host header; a missing Host fell through to a name-mismatch 404 rather than the RFC-required 400 Bad Request. Added an explicit check immediately after correlation ID generation: HTTP/1.1 requests without Host now return `400 invalid_request` before any routing or proxying. HTTP/1.0 is exempt per RFC 1945. Also added corpus case `tests/corpus/http/request/no_host_http11.http` and a unit test for the condition logic. 467 tests total. Closes #90.
- **Upstream `Server` and `X-Powered-By` headers leaked through proxy responses** — `shouldSkipUpstreamResponseHeader()` did not include `server` or `x-powered-by`, causing upstream technology headers to appear alongside Tardigrade's own `Server` header in proxy responses. Fixed by adding both to the skip list and adding a defense-in-depth filter call inside `writeBufferedUpstreamResponse()`'s header loop. 9 new unit tests. 465 tests total. Closes #89.
- **HSTS missing after deploy** — `TARDIGRADE_HSTS_ENABLED` was absent from the deployment env file, causing the new binary to default to `false` and omit `Strict-Transport-Security`. Fixed by adding `TARDIGRADE_HSTS_ENABLED=true` to the env file and ensuring it is enforced on all future deploys.

### Fixed
- Plain HTTP `proxy_pass` routes now reuse the shared upstream `std.http.Client` path instead of opening a one-off `Connection: close` socket for every proxied request, while preserving hop-by-hop header stripping semantics for upstream requests.
- Shared-client proxy requests now suppress Zig's default `User-Agent` header so upstreams receive the caller's forwarded `User-Agent` without a duplicate runtime-injected value.
- Access logging now reuses in-memory formatting buffers and writes directly to stderr on the unbuffered path, reducing per-request allocator churn on hot routes.
- Runtime JSON logs now use a fixed-buffer fast path with direct stderr writes for typical log lines, keeping long-message fallback behavior while reducing writer overhead.
- Benchmark runner `wrk` percentile parsing now stops at the first matching percentile line, avoiding malformed saved JSON on noisy output.
- Generic `proxy_pass` responses now keep upstream header metadata in a short-lived arena, reducing small per-request GPA allocations on proxied requests.
- Generic proxy request helpers now borrow existing `X-Forwarded-For` and upstream URL values when no rewrite is needed, avoiding no-op per-request allocations on the common path.
- The main HTTP/1 proxy route now uses a request-scoped temp arena for sticky-affinity and target-resolution scratch data, reducing short-lived allocator churn without changing upstream response ownership.
- Proxy request header assembly now uses stack-fallback buffers before spilling to the GPA, keeping ordinary header sets off the heap on the common path.
- Proxy response heads are now serialized into a stack buffer, and small buffered proxy responses now emit headers plus body in a single write when they fit, reducing small-write syscall overhead on the common HTTP/1 proxy path.
- The bundled benchmark upstream fixture now speaks explicit HTTP/1.1 keep-alive, and benchmark docs now require that fixture instead of `python -m http.server` for canonical proxy baselines.
- CI now installs OpenSSL development headers explicitly and enforces formatting consistently.
- Unix-socket upstream probing/proxying no longer depends on removed `std.http.Client.connectUnix()` behavior under Zig `0.16.0`; Unix-socket compatibility paths now fall back to raw request/response handling where needed.
- Proxy requests now strip RFC hop-by-hop headers, including headers named by the incoming `Connection` field, before forwarding to upstreams.
- Static file serving now rejects percent-encoded traversal, separator-variant traversal, and symlink escapes outside the configured root.
- Rate limiting now resolves authenticated identity before middleware enforcement so JWT, bearer, and session traffic do not share an IP-only bucket.
- Hot reload now retires superseded configs after in-flight requests complete so repeated reloads do not accumulate stale config allocations.
- Request IDs now accept `X-Request-ID` input, echo both request-id headers on responses, propagate upstream, and enrich JSON access logs with upstream address, upstream status, and response byte counts.
