### Testing
- **Failure-mode / chaos harness (#169)** â€” a dedicated production-safety suite (`zig build test-failure`) that intentionally breaks origins and clients, then asserts the gateway fails safely. Built on the existing live-process integration harness (mock origins + a real `tardigrade` process) as `failure:`-prefixed tests in `tests/integration.zig`, filtered into a standalone `test-failure` build step. Covers: origin down before connect, origin accepting-but-never-responding (bounded 5xx), origin closing mid-response (buffered path asserted to never fabricate the advertised body; streaming aborts already covered), malformed upstream response headers (bounded 5xx, never passed through as success), client aborting mid-upload (worker recovers) and mid-download (recorded via `tardigrade_proxy_client_aborts_total`), unreachable access-log sink (requests still succeed), and the metrics endpoint scraped under concurrent proxy load. Each scenario also asserts no worker starvation (an unrelated `/healthz` route stays responsive) and no leaked client connections (bounded `tardigrade_active_connections` gauge). Reload-during-active-streams and graceful-shutdown-under-load remain covered by the `#170` tests in the same file; the suite runs as part of `test-integration` in CI. Two findings surfaced by the harness are addressed in the same release: buffered-mode premature-upstream-close now returns 502 (#269) and TLS-handshake stall/failure coverage was added (#270).
- **TLS handshake chaos coverage (#270)** â€” extends the failure-mode harness with the deferred "TLS handshake stalls or fails" scenario. A generic TLS listener (enabled via `TARDIGRADE_TLS_CERT_PATH`/`_KEY_PATH`) is hit with a burst of malformed handshakes (raw non-TLS bytes and a truncated TLS record header, each closed immediately) plus one partial `ClientHello` held open for the rest of the test. The assertions: a valid `curl -k` request still terminates end-to-end â€” which, on the harness's single worker, proves the held-open stall is bounded by `TARDIGRADE_TLS_HANDSHAKE_TIMEOUT_MS` rather than pinning the worker â€” the `tls handshake error` log signal is emitted, and `tardigrade_active_connections` stays bounded (no leaked half-open handshakes).
- **CI perf smoke now guards upstream connection reuse (#141)** â€” `benchmarks/ci-smoke.sh` scrapes the gateway metrics endpoint after the proxy run and fails if fewer than 80% of upstream connections were reused (`reused / (reused + new)`). This catches a regression to per-request connections â€” the exact failure mode that motivated the pool (#196 dropped reuse to zero). The smoke fixture gains a `metrics_path`.

### Reliability
- **Buffered upstream premature-close now returns 502 instead of a truncated 200 (#269)** â€” when a proxied origin advertised a `Content-Length` (or chunked framing) but closed the connection before the body was complete, the buffered proxy path (`exchangeBoundedBufferedHttpRequest`) forwarded the bytes it had received and returned `200` with a short body â€” a silent truncation a downstream client could mistake for a complete response. The read loop now treats EOF before a `Content-Length`/chunked body reaches its declared end as `error.UpstreamProtocolError`, which the data-plane runtime maps to `502 Bad Gateway` (matching the streaming path, which already surfaces the abort). Close-delimited responses (no advertised length) and bodiless responses are unaffected. Surfaced by the failure-mode harness (#169) and now asserted by its buffered mid-response-close test.
- **Bounded TCP connect for FastCGI / SCGI / uWSGI upstreams (#171)** â€” the previous timeout pass bounded these protocols' request/response *exchange* but left the TCP `connect()` blocking, so a SYN-blackholed backend still pinned a worker for the kernel's ~2-minute limit. All three now route their TCP connect through `compat.connectBoundedTcp` (non-blocking connect + `poll` + `SO_ERROR`), bounded by `TARDIGRADE_UPSTREAM_CONNECT_TIMEOUT_MS` (default 5000); a timed-out connect surfaces as a 502. Unix-socket endpoints keep the local blocking connect (no SYN-blackhole exposure). Closes the "protocol-upstream TCP connect" gap documented in `docs/TIMEOUTS.md`; the remaining documented gaps are the cooperative request-total clock, downstream h2/h3 per-stream phases, and the soft shutdown drain.
- **Timeout policy: audit matrix + the three worst remaining unbounded waits closed (#171)** â€” new `docs/TIMEOUTS.md` documents every timeout class Tardigrade enforces (knob, default, mechanism, expiry behavior, honest status), refreshing the June audit against the post-#196/#141/#145 transports. Three genuinely unbounded worker-pinning waits found in the audit are fixed: **(1) upstream TCP connect** â€” `TARDIGRADE_UPSTREAM_CONNECT_TIMEOUT_MS` previously bounded only the handshake/write phases; the blocking `connect()` itself was uninterruptible, so a SYN-blackholed origin stalled a worker for the kernel's ~2-minute limit. `compat.connectBoundedTcp` (non-blocking connect + `poll` + `SO_ERROR`, blocking mode restored) now enforces it on the buffered, streaming, h2, and h2c paths; a timed-out connect maps to 504 `upstream_timeout`. **(2) h2/h2c pool TLS handshake** â€” the h2 pool handshook with no socket deadlines, so a TCP-accepting-but-silent origin hung the worker in `SSL_connect` indefinitely; the pool socket now gets `SO_RCVTIMEO`/`SO_SNDTIMEO` before the handshake (parity with the h1 paths). **(3) FastCGI / SCGI / uWSGI exchanges had no deadline at all** â€” a hung php-fpm/SCGI/uWSGI backend pinned a worker forever; all three now bound the request write and response reads with `TARDIGRADE_UPSTREAM_TIMEOUT_MS`. Tests: bounded-connect unit tests (blocking-mode restoration + round-trip; a saturated-backlog connect returns within the deadline instead of the kernel limit) and an h2-pool test against a TCP-accepting-but-silent TLS origin. Remaining documented gaps (cooperative request-total clock, downstream h2/h3 per-stream phases, protocol-upstream TCP connect, soft shutdown drain) are enumerated in the doc for follow-up PRs.
- **Event loop tolerates EINTR on kqueue (#145)** â€” the macOS/BSD `kevent` wait treated any failure as `error.Unexpected`, which surfaced as noisy "event loop wait error" logs once background worker threads (e.g. the new HTTP/2 upstream readers) started delivering signals that interrupt a blocking `kevent`. It now swallows `EINTR` and loops again (mirroring the epoll path, which already returned 0 on error), while still surfacing genuine failures. No functional impact on Linux.
- **FastCGI / SCGI / uWSGI no longer stall on the std.Io transport (#141, Phase 2)** â€” these protocols connected via the std.Io event-loop streams (`connectUnixSocket` / `tcpConnectToHost`), whose threaded backend can stall on a blocking request/response exchange (the same class of issue #196 settled for the data plane). They now use raw blocking sockets (`connectBlockingUnix` / `connectBlockingTcp` + `netStreamFromFd`). Verified end-to-end: FastCGI against php-fpm previously hung; it now returns correctly.
- **Bounded manual TCP/TLS transport for buffered upstream proxying (#196)** â€” the data-plane buffered proxy path (`executeBoundedBufferedHttpProxyRequest`) and the control-plane JSON proxy path no longer route through a shared `std.http.Client`, which did not expose the upstream socket fd and therefore silently ignored `TARDIGRADE_UPSTREAM_TIMEOUT_MS`, `TARDIGRADE_UPSTREAM_CONNECT_TIMEOUT_MS`, and `TARDIGRADE_UPSTREAM_RESPONSE_TIMEOUT_MS` on TCP and non-mTLS HTTPS upstreams â€” a hung TCP origin could block a worker indefinitely. Both paths now connect a plain blocking socket (`connectBlockingTcp` / `connectBlockingUnix`), optionally wrap it in TLS, and bound the response read with `poll(2)`. (`SO_RCVTIMEO` is honored on AF_UNIX sockets but silently ignored on the AF_INET upstream sockets created via the std.Io threaded backend, so `poll` is the authoritative read deadline.) Plain and mTLS HTTPS upstreams are now unified on the timeout-enforcing OpenSSL transport. The reader is framing-aware (Content-Length / chunked / bodiless / close-delimited) so it stops at the body boundary instead of waiting for EOF â€” required for keep-alive upstreams that hold the connection open. Also fixes a latent leak in `parseBufferedUpstreamResponse` where the response arena was copied into the returned struct before the in-literal `reason`/header allocations, leaking when a response carried only hop-by-hop headers.

### Performance
- **Upstream-pool Phase 1d completed: per-origin active cap, Unix-socket pooling, reuse benchmark (#239 / #141)** â€” the three deferred Phase 1d items land together. **(1) `TARDIGRADE_UPSTREAM_POOL_MAX_ACTIVE_PER_HOST`** (default 0 = unlimited) is now a real **fail-fast** hard cap on concurrently checked-out connections per origin: at the cap, requests are rejected with **503 `upstream_saturated`** instead of opening more connections. Fail-fast was chosen over queueing deliberately â€” in the thread-per-connection worker model, blocking until a slot frees would let one slow origin absorb the whole worker pool (the upstream-side worker-starvation tail documented on #141); queueing/watermark semantics are deferred to #140 where they couple with downstream admission control. Enforcement is race-free: the pool **reserves** the active slot under its mutex before the caller connects (`checkout`/`reserveSlot`/`releaseSlot`), so concurrent connects cannot exceed the cap. Saturation is a local policy rejection â€” it is **not** counted against passive upstream health or the circuit breaker, and does not burn the retry budget. New per-origin counter `tardigrade_upstream_pool_at_capacity_total{upstream}`. Applies to plain/TLS TCP, Unix-socket, and FastCGI checkouts (not the h2 pool, which is one multiplexed connection per origin bounded by `MAX_CONCURRENT_STREAMS`). **(2) Unix-socket upstream connections now pool** (keyed `unix:<path>`) on the buffered data-plane and control-plane paths, with the same keep-alive framing rules, stale-retry-once, idle/lifetime eviction, and metrics as TCP â€” previously every unix request opened a fresh `Connection: close` connection. **(3) `benchmarks/upstream-reuse.sh`** formalizes the reuse scenario (reuse ratio, new-connection count, local/cross-worker split, stale retries) and adds an active-cap demonstration phase against a new `/slow?ms=` fixture endpoint (503s at saturation, `at_capacity_total`, follow-up request proving health is untouched); the â‰¥80% reuse CI gate continues to live in `benchmarks/ci-smoke.sh`.
- **Cleartext h2c upstreams via prior knowledge (#237 / #145)** â€” `TARDIGRADE_UPSTREAM_PROTOCOL=h2c` makes plain-HTTP (`http://`) upstreams speak HTTP/2 with prior knowledge (RFC 9113 Â§3.3): the h2 preface goes out immediately on the plain socket â€” no ALPN, and no deprecated HTTP/1.1 `Upgrade: h2c` dance. Cleartext has no negotiation, so this is a deliberate explicit opt-in: `h2`/`auto` never imply it and existing plain-HTTP/1.1 upstreams are untouched (for HTTPS upstreams `h2c` behaves exactly like `h2`). h2c connections share the whole TLS-h2 machinery â€” the production connection type is now `H2Conn(*UpstreamH2Transport)`, a runtime union over the OpenSSL connection and a plain socket â€” so multiplexing, the per-origin pool, refcounted lifecycle, idle/lifetime eviction, TCP_NODELAY, the buffered *and* streaming proxy paths, bounded streaming backpressure, and all h2 metrics apply unchanged; h2c origins appear under `h2c:host:port` keys in the per-origin series and count as `protocol="h2"`. Verified end-to-end against `nghttpd --no-tls`: buffered and streaming relays byte-for-byte (70 KB / 1.2 MB), ~9.2k `wrk` requests multiplexed over **one** cleartext upstream connection with zero non-2xx; the `http1` default still speaks HTTP/1.1 to plain upstreams (it fails with a clean 502 against this h2-only origin and is untouched for ordinary h1 origins, which the existing suite covers). Deferred within the h2 arc: streaming uploads over h2.
- **Per-origin HTTP/2 upstream metrics (#238 / #145)** â€” the h2 upstream series are now labelled by origin: `tardigrade_upstream_h2_pool_connections_active` / `â€¦_streams_active` (gauges) and `â€¦_stream_resets_total` / `â€¦_goaway_total` (counters), all keyed `{upstream="h2:host:port"}` (the pool key, matching the scheme-prefixed h1 label convention), so operators can identify *which* origin is producing resets or GOAWAYs. Internally each `H2Conn` reader now bumps a persistent per-origin counter struct instead of the two pool-level atomics (same bump-before-publish ordering); the entries are created on first h2 connection to an origin and never removed while the pool lives, so an evicted origin keeps reporting its counters (with zeroed gauges) and the totals stay monotonic. The existing global series (`tardigrade_upstream_h2_connections_active` / `â€¦_streams_active` / `â€¦_stream_resets_total` / `â€¦_goaway_total`) are unchanged and backward-compatible â€” the counter globals are now sums over the per-origin entries with identical values. The `_pool_` name prefix (mirroring the labelled h1 `tardigrade_upstream_pool_*` series) keeps a label-blind `sum()` over the labelled family from double-counting the bare globals. Verified end-to-end against two `nghttpd` origins: independent labelled series per origin, correct gauge transitions across connection eviction and reconnect, and label persistence after an origin died. Together with `tardigrade_upstream_request_latency_ms{protocol}` (shipped with the streaming-path PR), this completes #238's scope.
- **Streaming proxy path over HTTP/2 upstream multiplexing (#141, Phase 4b PR 4 / #145)** â€” the streaming reverse-proxy path (`TARDIGRADE_PROXY_STREAMING_MODE=response|full`) now multiplexes over the shared per-origin HTTP/2 upstream connection when `TARDIGRADE_UPSTREAM_PROTOCOL=h2|auto`, instead of always speaking HTTP/1.1. The `H2Conn` actor gains a streaming mode (`requestStreaming` / `readStreamingBody` / `finishStreaming`): the reader thread parks DATA frames in a bounded per-stream buffer and â€” unlike the buffered path, which replenishes flow-control windows immediately â€” replenishes the **stream-level** window only as the relay drains downstream, so the advertised 1 MiB initial stream window *is* the buffer bound and a slow downstream client backpressures its own stream at the origin (a window-overrunning peer is failed with `Http2FlowControlError` rather than buffered without bound). The **connection-level** window is always replenished promptly (and grown to 8 MiB at connection start), so one slow client cannot stall the other streams sharing the connection â€” measured: a 10 KB/s client mid-download did not move the ~7 ms latency of 40 concurrent fast requests on the same single upstream connection. Abandoned/aborted relays RST_STREAM their stream so the origin stops sending; response trailers are decoded (required for connection-wide HPACK dynamic-table consistency â€” previously a header block for an already-completed stream was skipped, which would desynchronize the decoder) and dropped, matching the h1 relay. Every worker wait on the shared connection is now deadline-bounded even while frames keep flowing for other streams (a reader-side sweep fails no-progress streams with `Http2Timeout`; progress extends the deadline â€” this also closes a latent starvation window in the buffered `request()` wait), and the request-body sender no longer holds the frame write mutex while waiting for send window (a latent deadlock: the reader needs that mutex for PING/SETTINGS acks; DATA frames from concurrent streams may now interleave). Also fixes reader-side signal-after-unlock races on per-stream condition variables and END_STREAM getting lost on a CONTINUATION-completed HEADERS block. Adds `tardigrade_upstream_request_latency_ms{protocol="h1"|"h2"}` (histogram of completed upstream exchanges â€” the last #145 metrics row, "upstream p99 by protocol"), and the streaming path now counts toward `tardigrade_upstream_protocol_requests_total`. Verified end-to-end against `nghttpd`: 70 KB / 200 KB / 1.2 MB bodies relayed byte-for-byte through the streaming h2 path (past both the 64 KiB default and 1 MiB advertised windows), ~31k `wrk` requests at ~3.1k req/s over **one** upstream connection with zero non-2xx and stable fds, slow+fast client isolation as above, and byte-correct ALPN `http/1.1` fallback and plain-HTTP h1 streaming. Streaming **uploads** (`full` mode request bodies) stay on the HTTP/1.1 path (deferred with cleartext h2c).
- **HTTP/2 upstream hardening: reset/GOAWAY metrics, idle eviction, TCP_NODELAY (#141, Phase 4b PR 3 / #145)** â€” completes the h2 upstream pool. **(1) Metrics:** two new counters surface protocol-level failures â€” `tardigrade_upstream_h2_stream_resets_total` (RST_STREAM frames received) and `tardigrade_upstream_h2_goaway_total` (GOAWAY frames received). These are pool-level monotonic atomics bumped by each connection's reader thread (via a pointer handed to `H2Conn.init`), so the totals **survive connection teardown** â€” a live-connection snapshot would lose them when an evicted conn is freed. **(2) Idle/lifetime eviction:** `H2ConnPool.reapIdle` now runs on the gateway maintenance tick (alongside `upstream_pool.reapIdle`), closing h2 connections that are idle past `TARDIGRADE_UPSTREAM_POOL_IDLE_TIMEOUT_MS`, past `TARDIGRADE_UPSTREAM_POOL_MAX_LIFETIME_MS`, or already dead (GOAWAY/errored). Eviction is refcount-safe: a connection with in-flight streams is never reaped (`activeStreamCount() == 0` gate), and the `release` that may tear a connection down runs *after* the pool mutex is dropped so a teardown never blocks the pool; each connection stamps `last_activity_ms` when a stream starts and finishes. **(3) TCP_NODELAY:** the h2 upstream socket now disables Nagle â€” h2 multiplexing issues many small frame writes (HEADERS / WINDOW_UPDATE) whose interaction with the peer's delayed ACK otherwise stalled each exchange ~40 ms and tripped response timeouts under concurrency. Verified end-to-end against `nghttpd` on Linux: enabling NODELAY took the buffered h2 proxy path from **24 req/s â†’ 5.4k req/s** (single-request latency 35 ms â†’ 0.7 ms) with the whole concurrent load multiplexed over **one** upstream connection, zero client-visible 5xx, and stable fds across repeated idle-reap cycles; the reset/GOAWAY counters and idle reaper were exercised directly. Adds `benchmarks/h1-vs-h2-upstream.sh` (h1 pool vs h2 multiplexed). Deferred: the streaming proxy path over h2 and cleartext h2c (later work).
- **HTTP/2 upstream multiplexing (#141, Phase 4b PR 2 / #145)** â€” many concurrent proxied requests now share a single upstream HTTP/2 connection. A per-connection actor (`H2Conn` in `src/http/upstream_h2.zig`) runs a dedicated reader thread that owns all socket reads and HPACK decoding (the dynamic table is connection-wide, so exactly one decoder), while worker threads call a blocking `request()` that opens a stream, writes HEADERS/DATA under a write mutex, and waits on a **per-stream** condition until the reader delivers its response. Connections are held in a refcounted per-origin pool (`H2ConnPool`, one connection per origin, keyed `h2:host:port`) so an evicted/dead connection stays alive until its last in-flight request drains; a connection-level failure evicts the connection and retries once on a fresh one. Full connection + stream flow control (WINDOW_UPDATE, INITIAL_WINDOW_SIZE deltas), CONTINUATION, `MAX_CONCURRENT_STREAMS` backpressure, and GOAWAY/RST_STREAM handling are implemented. New metrics: `tardigrade_upstream_h2_connections_active` and `tardigrade_upstream_h2_streams_active` (gauges). Verified end-to-end against `nghttpd`: **32 concurrent clients served ~98k requests over a single upstream h2 connection at ~12k req/s** (p50 331 Âµs), byte-correct, zero errors. Notable fix during bring-up: an initial single broadcast-condition design caused a thundering herd that collapsed throughput under concurrency (âˆ’c32 fell to ~477 req/s); switching to per-stream condition variables restored it to ~12k req/s (â‰ˆ25Ã—). Deferred to the next PR: reset/GOAWAY counters, idle/lifetime eviction of h2 connections, the streaming proxy path, and h1-vs-h2 benchmarks. TLS only (ALPN); cleartext h2c is not supported.
- **HTTP/2 upstream foundation: ALPN + single-stream h2 client (#141, Phase 4b PR 1 / #145)** â€” first step toward HTTP/2 upstream multiplexing, built entirely on the in-repo h2 stack (`http2_frame` + `hpack`), no external library. Upstream HTTPS connections can now negotiate HTTP/2 via ALPN: set `TARDIGRADE_UPSTREAM_PROTOCOL=h2` (or `auto`; default `http1`) and the buffered proxy path offers `h2,http/1.1`, uses HTTP/2 when the origin selects it, and transparently falls back to HTTP/1.1 otherwise. The new `src/http/upstream_h2.zig` performs a single request/response per connection (preface + SETTINGS + HEADERS/DATA, reading frames until the stream ends) with full connection/stream flow control (WINDOW_UPDATE), CONTINUATION, padding/priority, and SETTINGS/PING/GOAWAY/RST_STREAM handling; reads are `poll(2)`-bounded (plus `SSL_pending`) so a hung origin cannot block a worker. - **Cross-worker upstream-pool reuse: analysis, metrics, and benchmark (#141, Phase 4a / #147)** â€” confirmed that Tardigrade's single shared `UpstreamPool` already reuses idle upstream connections *across worker threads* rather than fragmenting an idle pool per worker (the Pingora/HAProxy failure mode #147 targets). - **Streaming proxy path off `std.http.Client`, now pooled and timeout-bounded (#141, Phase 3 / #196)** â€” the streaming reverse-proxy path (`executeStreamingHttpProxyRequest`, used when `TARDIGRADE_PROXY_STREAMING_MODE=response|full`) no longer routes through a shared `std.http.Client`. It now uses the same manual bounded TCP/TLS transport and shared upstream connection pool as the buffered path: the upstream socket fd is owned by Tardigrade, so per-phase timeouts (`TARDIGRADE_UPSTREAM_CONNECT_TIMEOUT_MS` / `â€¦_RESPONSE_TIMEOUT_MS`) are now enforced via `poll(2)` on the streaming path too â€” closing the last #196 timeout gap (a hung streaming origin could previously block a worker indefinitely). The upstream response head and body are parsed and relayed with a manual, framing-aware reader (Content-Length / chunked / bodiless / close-delimited) that decodes the upstream body and re-chunks it downstream, with bounded buffering (`TARDIGRADE_PROXY_STREAM_BUFFER_SIZE`, min 16 KiB) and cancellation / client-abort handling. ### Performance
- **Idle keepalive connections no longer occupy a worker thread (#138)** â€” Tardigrade serves requests on a bounded worker pool with blocking I/O. Previously a worker stayed attached to a keepalive connection for its entire lifetime (blocked in `read()` between requests), so once concurrent connections exceeded the worker count the surplus connections starved and tail latency exploded. After serving a request, a keepalive connection is now parked off the worker pool: its state (fd, pooled session, optional TLS session) moves into a registry, its fd is watched by the event loop, and the worker returns to the pool. When the next request arrives the event loop hands the connection back to a worker, which serves one request and re-parks or closes. Active request handling still uses a worker with blocking I/O, so only the idle gap is decoupled. Measured on the 4-core deployment hardware at the default `worker_threads` = CPU count: **p90 dropped from 26.5 ms to 1.33 ms at 10 connections (20Ã—), and from 412 ms to 6.2 ms at 100 connections (66Ã—)**, with throughput up 10â€“27%.
